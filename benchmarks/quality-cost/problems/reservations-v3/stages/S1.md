# Stage S1 — Room reservation service (interval correctness)

Build an HTTP/JSON service for reserving rooms over half-open integer time intervals `[start, end)`.
Any language/structure (standard library only; Node v25 and python3 available). Honor the run
contract (`start.sh`, foreground, `$PORT`, prints `LISTENING <port>`).

## Endpoints

- `POST /rooms` body `{ "id": <string> }`
  - 201 `{ id }`; 400 if `id` missing/empty; 409 if the room already exists

- `POST /rooms/:id/reserve` body `{ "start": <int>, "end": <int>, "holder": <string> }`
  - 201 `{ reservationId, roomId, start, end, holder, state: "active" }` (reservationId any unique string)
  - 404 if the room does not exist
  - 422 if `start`/`end` are not integers or `start >= end`
  - 409 if the interval `[start, end)` overlaps any **active** reservation on that room

- `POST /reservations/:rid/cancel`
  - 200 `{ reservationId, state: "cancelled" }`; idempotent (cancelling twice still 200)
  - 404 if the reservationId is unknown
  - a cancelled reservation frees its interval (it may then be reserved again)

- `GET /rooms/:id/reservations`
  - 200 `{ roomId, reservations: [ { reservationId, start, end, holder, state } ] }`
  - 404 if the room does not exist

## The overlap rule (read carefully)
Intervals are **half-open** `[start, end)`. Two intervals `[a,b)` and `[c,d)` **overlap** iff
`a < d AND c < b`. Therefore intervals that merely **touch** at an endpoint do **not** overlap:
`[10,20)` and `[20,30)` are both allowed on the same room. But `[10,20)` and `[15,25)` overlap,
as do containment cases like `[10,30)` vs `[15,20)`.

## Invariant you MUST preserve
On every room, no two **active** reservations overlap, at all times.

Include tests — especially the adjacency boundary, partial overlap, containment, and
reserve-after-cancel cases.
