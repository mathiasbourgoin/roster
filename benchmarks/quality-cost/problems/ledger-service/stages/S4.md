# Stage S4 — Transfers (complex, error-prone, invariant-stressing)

Add atomic transfers between two accounts. Keep all S1–S3 behavior working.

- `POST /transfer` with body `{ "from": "<id>", "to": "<id>", "amount": <number> }`
  - 200 with `{ "from": { "id", "balance" }, "to": { "id", "balance" } }`
  - 404 if either account does not exist
  - 422 if `amount` is not a number > 0
  - 422 if `from === to`
  - 422 if `from` has insufficient funds

**Critical invariant (do not violate):** a transfer must be **atomic** and must
**conserve total money** — the sum of all account balances is unchanged by a transfer,
no balance ever goes negative, and a rejected transfer leaves BOTH accounts exactly as
they were (no partial debit/credit). This is the error-prone part: a naive
debit-then-credit that can fail in the middle will corrupt the ledger.

Add tests, including concurrent/failed transfers that prove no money is created or lost.
