# Stage S4 — Cross-cutting: idempotent creates + optimistic-concurrency versioning

Add two cross-cutting capabilities. **This must NOT break any existing S1–S3 behavior** — clients
that don't use the new features must keep working exactly as before (backward compatible).

## Versioning
- Every task now has an integer `version`, starting at **1** when created.
- `version` is included in the JSON of `GET /tasks/:id`, `POST /tasks`, and the list `items`.
- Each **successful status change increments the task's `version` by exactly 1**.

## Idempotent creates
- `POST /tasks` now honors an optional request header `Idempotency-Key`.
  - Two `POST /tasks` with the **same** `Idempotency-Key` return the **same** task (same `id`) and
    do NOT create a duplicate.
  - No header (or a different key) → a new task as before.

## Optimistic concurrency on status change
- `POST /tasks/:id/status` now accepts an optional body field `expectedVersion`.
  - If `expectedVersion` is provided and does NOT equal the task's current version → **409**, no change.
  - If omitted → behave exactly as in S2 (backward compatible).

Add tests for idempotent replay, version increments, and the stale-version conflict.
