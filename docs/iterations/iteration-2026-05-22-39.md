# Iteration 2026-05-22-39: Selected Agent Start

## Brief

Loop 39 implements the first TUI-first start action. The MIAOU dashboard can now
start the selected agent through the local TA socket instead of requiring a
whole-workspace launch command.

## Acceptance Criteria

- Add a socket `start-agent` request for one workspace agent.
- Keep `start-agent` launch commands and actor identity server-trusted: clients
  choose only the workspace and agent; the socket server owns the config path
  and bound actor.
- Reuse launch planning so selected-agent starts use the same typed config,
  cwd, environment, prompt, pane id, and tmux target rules as supervised
  workspace launches.
- Require write authorization from the actor before starting the selected agent.
- Reject already attached agents before launch and again before state update.
- Clean up the created tmux pane or session if post-launch state persistence
  fails.
- Bind `s` in the MIAOU TUI to start the selected agent and refresh the
  dashboard on success.
- Keep direct config dashboards read-only for start actions.
- Validate with OCaml tests, MIAOU headless execution, and isolated tmux smoke.

## Result

Completed.

- Added `Launch_plan.find_agent` and `Launch_state.preflight_agent`.
- Added `Launch_runtime.run_agent_with` plus single-pane cleanup support.
- Added `Tmux.Kill_pane` for split-pane cleanup.
- Added socket protocol and CLI support for `start-agent`, with config and actor

- Wired `ta --socket ... --tui=always` so pressing `s` starts the selected

- Marked the initial TUI dashboard as freshly loaded and preserved runner

  headless idle ticks.
- Added tests for selected-agent launch planning/runtime, state preflight,
  socket CLI validation, and TUI stale-action feedback.

## Review Notes

The selected-agent path intentionally does not filter and reuse whole-plan
`launch start`. Whole-plan launch refuses existing sessions and owns cleanup at
workspace granularity, while a TUI action must be able to split into an existing
workspace session and clean up exactly one pane.

`start-agent` still requires the socket server to be launched with a trusted
config and bound actor because the state snapshot does not persist
command/cwd/env/startup prompt data or authenticated actor sessions. A future
loop should remove the need for users to know those details by making the
installed harness/config the primary TUI source.

Generated `index.json` changed during `npm test` and is intentionally left out
of the commit set for this loop.
