# Iteration 2026-05-22-12: Native Pane Targeting

## Brief

Remove the runtime assumption that managed tmux windows and panes use default
indexes. TA should target launched agents by native `%pane_id`, not by
`session:0.N`.

## Acceptance Criteria

- Capture pane ids from tmux creation commands with `-P -F "#{pane_id}"`.
- Send startup prompts to captured native pane ids.
- Keep dry-run useful even though prompt targets are only known at runtime.
- Preserve launch-state attachment behavior.
- Verify behavior with non-default `base-index` and `pane-base-index`.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added tmux command variants for `new-session -P -F "#{pane_id}"` and
  `split-window -P -F "#{pane_id}"`.
- Reworked `Launch_runtime.run_with` to capture native pane ids during launch
  and use those ids for attachments and startup prompts.
- Changed layout targeting from `session:0` to the session target so it is not
  tied to window index zero.
- Added `Launch_runtime.dry_run_lines`, which preserves executable command
  rendering and adds explicit prompt placeholder comments for agents with
  startup prompts.
- Added deterministic tests for captured pane attachments, prompt targeting,
  split failure cleanup, prompt failure cleanup, and dry-run prompt previews.
- Reviewer found a dry-run prompt preview regression; it was remediated before
  commit and reviewer recheck passed.
- QA verified a live launch under non-default tmux indexes where window index
  was not zero and pane indexes were not zero-based.

## Next Loop Candidate

Add the Unix socket API boundary mirroring safe `tactl` state and launch
commands. That gives the future TUI and MCP bridge a stable local control plane
without binding them directly to CLI process execution.
