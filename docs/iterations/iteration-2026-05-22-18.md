# Iteration 2026-05-22-18: Dashboard Foundation

## Brief

Start the MIAOU/TUI dashboard foundation without coupling UI widgets directly
to state files, tmux polling, or socket transport. The loop creates a typed
view model and a static frame renderer so future MIAOU screens can focus on
interaction, layout, and keyboard flow.

## Acceptance Criteria

- Add a pure dashboard model that merges `State_store` and
  `Runtime_snapshot`.
- Render a dense workspace/agent frame with runtime health, ACL summaries, and
  a pane preview panel.
- Add `tactl dashboard render STATE` for deterministic QA snapshots.
- Upgrade `ta --state STATE` so the main launcher shows the same dashboard
  frame.
- Cover the model, `tactl`, and `ta` paths with tests.
- Run real tmux QA against a launched workspace.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added `Dashboard_model`, a backend-neutral model for workspace summaries,
  agent rows, outgoing read/write connections, runtime state, and previews.
- Added a deterministic static renderer that gives TA a first dashboard frame
  while keeping the eventual MIAOU renderer thin.
- Added `tactl dashboard render --width COLS --lines N STATE`.
- Updated `ta` so `ta --state STATE` renders the dashboard frame; config-only
  usage still prints the bootstrap summary.
- Added model tests, `tactl dashboard` CLI tests, and `ta --state` CLI tests.
- QA launched a disposable two-agent tmux workspace, rendered a live dashboard,
  verified live pane state, ACL text, and preview output, and checked the `ta`
  launcher path.

## Next Loop Candidate

Add the first interactive MIAOU runner around `Dashboard_model`: keyboard
selection across workspaces/agents, refresh ticks, and a renderer boundary that
can later swap local state loading for socket-backed refresh.
