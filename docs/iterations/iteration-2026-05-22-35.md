# Iteration 2026-05-22-35: First Full-Screen Dashboard TUI

## Brief

Loop 34 fixed first-run startup and help. Loop 35 shifts the roadmap toward
the Herdr-quality center of gravity: TA must be TUI-first, not a CLI that
happens to render dashboard text. This loop introduces the first real
alternate-screen interactive terminal loop while preserving deterministic static
output for tests, pipes, and automation.

## Acceptance Criteria

- Add a full-screen dashboard mode for `ta` on real terminals.
- Keep non-TTY `ta` output static and scriptable.
- Add `--tui=auto|always|never`:
  - `auto` enters the TUI only when stdin/stdout are terminals;
  - `always` rejects non-terminal execution with a clear error;
  - `never` forces one static frame.
- Use existing dashboard interaction, runner, and refresh behavior so keyboard
  commands stay consistent with static `--key` replay.
- Ensure terminal ownership is released on quit and exceptions.
- Cover static fallback, forced-static mode, forced-TUI non-terminal rejection,
  and help output.
- Run pseudo-terminal QA that starts the TUI, captures a dashboard frame, quits
  with `q`, and confirms cleanup.
- Run review, QA, commit, and push the loop branch.

## Result

Completed.

- Added a Notty-backed full-screen runner for `ta`.
- `ta --tui=auto` keeps redirected output static but enters alternate-screen
  TUI mode on a real terminal.
- `ta --tui=always` now provides an explicit terminal requirement check.
- `ta --tui=never` preserves deterministic one-frame rendering.
- The TUI keeps `Dashboard_runner.t` as state and feeds periodic tick events,
  preserving automatic refresh cadence without requiring key input.
- Removed an unimplemented `? help` footer binding from the first TUI loop.
- Added tests for the new CLI mode contract.
- Verified pseudo-terminal startup/quit and no-key auto-refresh through tmux.
