# Iteration 2026-05-22-5: State Snapshot JSON

## Brief

Make the in-memory state store durable enough for the future daemon by adding a
versioned JSON snapshot format. This is still a pure library slice: no daemon
process and no filesystem persistence command yet.

## Acceptance Criteria

- Add `State_store.to_yojson` and `State_store.of_yojson`.
- Snapshot workspaces, agent status, pane attachment, ACL links, audit events,
  and next sequence number.
- Reject unsupported snapshot versions with structured errors.
- Add tests for state round-trip and bad version rejection.
- Keep files within the project size limits by splitting state-store tests.
- Run review, QA, and tmux smoke before commit.

## Result

Completed.

- Split shared runtime records into private `State_model` and private
  `State_snapshot` modules while keeping `State_store` as the public API.
- Added versioned JSON snapshots for workspaces, agent status, panes, ACL
  links, audit events, and `next_seq`.
- Added graph validation on restore for duplicate workspaces/agents, empty
  link permissions, missing link endpoints, and invalid audit references.
- Tightened audit sequence validation so `next_seq` must equal the last audit
  sequence plus one.
- Expanded tests for round-trip continuity, unsupported versions, stale/future
  `next_seq`, audit sequence gaps, broken audit references, and broken graph
  links.
- Review initially found the future `next_seq` gap bug; it was fixed and
  re-reviewed with no remaining findings.
- QA passed OCaml build/tests/format/lint, roster-aware config validation,
  tmux smoke execution, and root roster tests.
