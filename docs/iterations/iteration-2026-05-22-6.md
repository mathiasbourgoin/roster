# Iteration 2026-05-22-6: State Snapshot Files

## Brief

Add the first filesystem persistence boundary for TA runtime state. The daemon
does not exist yet, so this loop focuses on deterministic snapshot save/load
commands that future daemon and TUI code can reuse.

## Acceptance Criteria

- Add a small core module for snapshot file save/load with structured errors.
- Add `tactl state save --output STATE CONFIG` to create an initial state
  snapshot from a validated workspace config.
- Add `tactl state load STATE` to validate and summarize a state snapshot.
- Cover the file boundary and CLI commands with tests.
- Run review, QA, and tmux smoke before commit.

## Result

Completed.

- Added `State_file` for atomic snapshot save and validated load with
  structured IO, JSON, and semantic snapshot errors.
- Added `State_store.summarize` so command surfaces can report restored state
  without exposing internal records.
- Added `tactl state save --output STATE CONFIG` and `tactl state load STATE`.
- Added core file-boundary tests and command-level save/load tests.
- Review found two low issues: missing fixture dependency tracking and missing
  filename context for semantic snapshot errors. Both were fixed and
  re-reviewed with no remaining findings.
- QA passed OCaml build/tests/format/lint, manual `tactl state` save/load,
  roster-aware config validation, tmux smoke execution, and root roster tests.
