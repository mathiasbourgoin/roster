# Iteration 2026-05-22-42: Harness Workspace Startup

## Brief

Loop 42 removes the next first-use trap: this repository has a canonical
`.harness/harness.json`, but a bare `ta` could still fall through to the bundled
example when `.harness/ta.json` was absent. TA should be workspace-enabled from
the roster harness itself.

## Acceptance Criteria

- Resolve startup defaults in this order: saved state, TA workspace config,
  canonical agent-roster harness, bundled examples.
- When `.harness/harness.json` exists and no TA config exists, derive a
  launchable `.harness/ta.json`.
- Bootstrap `.ta-state.json` from the derived config and open the real workspace
  dashboard.
- Prefer `tech-lead` as the initial selected agent when it exists.
- Keep help and quickstart text clear that a harness can be enough to start TA.
- Cover the projection and startup path with OCaml tests and MIAOU headless
  evidence.

## Result

Completed.

- Added `Ta_core.Harness_ta_config`, a typed projection from
  `.harness/harness.json` into a TA workspace config.
- Added `Workspace_config.to_yojson`/`to_string` so generated TA configs use the
  same structured model as parsed configs.
- Startup resolution now selects the canonical harness before bundled examples.
- Bare `ta` can generate `.harness/ta.json`, create `.ta-state.json`, and render
  the real harness workspace.
- Generated harness configs order `tech-lead` first, then `recruiter`, then the
  remaining harness agents in source order.
- Dashboard initialization also prefers `tech-lead` when present.
- Existing `.harness/ta.json` and `ta.json` still win over harness derivation.
- Malformed harness JSON stops startup with a concrete error instead of falling
  through to the bundled examples.
- Generated `.harness/ta.json`, `.ta-state.json`, and lock files are ignored so
  trying TA from the repo root does not dirty Git.

## Validation

- `opam exec -- dune build @all @runtest`
- MIAOU headless temp-workspace test: `.harness/harness.json` only, no TA config,
  renders `agent-roster`, selects `tech-lead`, shows `s Start tech-lead`, and
  creates both `.harness/ta.json` and `.ta-state.json`.
- Repo-root tmux smoke: full MIAOU TUI rendered the real 12-agent harness
  workspace and selected `agent-roster/tech-lead`.

## Review Notes

The generated config deliberately uses `codex` as the default command and a
simple coordinator ACL: `tech-lead` and `recruiter` get read/write links to
other harness agents. Dashboard initialization also prefers `tech-lead` when it
exists. The next loop should expose command/profile and creation authority more
explicitly in the TUI instead of leaving that power implicit in generated JSON.

Generated `index.json` remains unstaged.

## Next Loop Candidate

Add a TUI-visible harness provenance and privilege panel: show that the current
workspace came from `.harness/harness.json`, show the generated config path,
and mark `tech-lead`/`recruiter` as agents allowed to create or connect other
agents.
