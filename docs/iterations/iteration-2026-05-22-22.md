# Iteration 2026-05-22-22: Dashboard Refresh Cadence

## Brief

Add the pure refresh cadence policy the future MIAOU runner needs before it has
a terminal event loop. The prior loop made manual refresh executable through the
socket; this loop adds deterministic tick decisions, retry throttling, stale
thresholds, and user-visible last-refresh metadata.

## Acceptance Criteria

- Add a pure refresh cadence module with typed seconds and timestamps.
- Support interval-based refresh decisions.
- Support manual refresh override.
- Throttle automatic retries after failures.
- Detect stale dashboards after a configured threshold.
- Expose last success, last attempt, failure count, and a status line.
- Render last-refresh metadata in dashboard frames when captured runtime time is
  available.
- Cover cadence decisions and metadata rendering with tests.
- Run real tmux QA because the rendered dashboard frame changed.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added `Dashboard_refresh_cadence`, a pure policy module with typed `seconds`
  and `timestamp` values.
- The cadence decides between `Refresh reason` and `Wait duration` for manual
  keys, first refreshes, elapsed intervals, stale dashboards, and retry
  throttling.
- The module records success/failure metadata and renders a compact status line
  for future UI use.
- `Dashboard_interaction.render` now includes relative `Last refresh: ...`
  metadata when the dashboard model has a captured runtime timestamp.
- Added unit tests for manual override, interval wait/refresh, stale threshold,
  retry throttling, status lines, and invalid policy validation.
- QA launched a disposable tmux workspace, refreshed through the socket, and
  verified the last-refresh line plus selected live QA preview.
