# Iteration 2026-05-22-13: Read-Only Unix Socket API

## Brief

Add the first local control-plane boundary so the future TUI and MCP bridge can
query TA state without shelling out to `tactl`.

## Acceptance Criteria

- Add a line-delimited JSON request/response protocol.
- Serve state snapshot summary and detailed state inspection over a Unix socket.
- Add `tactl socket serve` and `tactl socket request` commands.
- Provide a `--once` server mode for deterministic QA.
- Avoid unsafe socket path deletion and contain malformed client behavior.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added `Socket_protocol` with typed `state-summary` and `state-show` requests
  and success/failure responses.
- Added `Socket_api` with a Unix socket server and client.
- Added `tactl socket serve --socket SOCKET --state STATE [--once]`.
- Added `tactl socket request --socket SOCKET state-summary`.
- Added `tactl socket request --socket SOCKET [--audit-limit N] state-show`.
- Hardened serving so existing regular files are not unlinked, stale sockets
  are removed safely, request lines have a timeout and maximum size, and
  per-client I/O failures do not terminate long-running serving.
- Added end-to-end CLI tests for summary, show, unknown command, negative audit
  limit, and refusing regular-file socket paths.
- Reviewer found socket path deletion and blocking-client risks; both were
  remediated before commit and reviewer recheck passed.
- QA verified socket summary/show requests through disposable sockets and
  confirmed cleanup.

## Next Loop Candidate

Extend the socket control plane with ACL-checked state mutations first, then
launch requests. Mutations should reuse existing `State_store` transitions and
preserve audit events rather than introducing socket-only behavior.
