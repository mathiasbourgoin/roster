# Iteration 2026-05-22-44: Typed Agent Capabilities

## Brief

Loop 44 makes TA explicit about which roster agents may eventually create new
agents or connect existing agents. The feature is metadata-only in this loop:
it should make authority visible without adding extra setup steps or new hidden
runtime behavior.

## Product Constraint

TA must stay Herdr-simple in the normal path. Capability metadata can explain
why `tech-lead` or `recruiter` get future create/connect actions, but starting
the app should still be workspace, agent, action.

## Acceptance Criteria

- Add a typed `Agent_capability` module with `create-agent` and
  `connect-agents` parsing/serialization.
- Allow optional per-agent `capabilities` in `.harness/ta.json`, defaulting to
  none when omitted.
- Persist capabilities through state snapshots while keeping older snapshots
  readable when the field is absent.
- Derive harness TA configs that grant both capabilities only to `tech-lead`
  and `recruiter`.
- Propagate capabilities into dashboard and MIAOU models.
- Render selected-agent capabilities as a compact detail row.
- Keep read/write ACLs and socket start behavior unchanged.

## Result

Completed.

- Added `Ta_core.Agent_capability` with phantom witness types, parser,
  serializer, equality, ordering, and grant helpers.
- Threaded capabilities through workspace config, state model/store, snapshot
  JSON, dashboard model, static layout, and MIAOU view.
- Harness projection grants `create-agent,connect-agents` to `tech-lead` and
  `recruiter`; other agents remain `none`.
- MIAOU selected-agent detail now shows `Capabilities create-agent,connect-agents`
  or `Capabilities none`.
- Schema and roadmap now document that capabilities are product authority,
  separate from session read/write ACLs.

## Review Notes

This loop intentionally does not implement create/connect actions or
enforcement. Those should be separate TUI flows that preserve the two-selection
startup constraint.

Generated `index.json` remains unstaged.

## Next Loop Candidate

Turn these capabilities into one simple TUI action surface: privileged agents
see create/connect actions, while non-privileged agents see request/disabled
states without exposing configuration-heavy forms.
