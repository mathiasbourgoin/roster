# Iteration 2026-05-22-43: Harness Provenance In The TUI

## Brief

Loop 42 made TA derive a workspace from `.harness/harness.json`. Loop 43 makes
that behavior visible so the app stays simple without hiding where authority
and generated state came from.

## Acceptance Criteria

- Preserve `harness_path` from `Workspace_config` into durable TA state
  snapshots.
- Surface the selected workspace source in static dashboard output.
- Surface harness provenance in the MIAOU selected-agent detail panel.
- Surface a compact privilege summary for the selected agent, based on visible
  read/write ACL edges.
- Keep old state snapshots readable when they do not have `harness_path`.
- Cover state preservation and MIAOU frame visibility in tests.

## Result

Completed.

- Added optional `harness_path` to `State_model.workspace`, state snapshot
  JSON, `State_store`, and `Dashboard_model`.
- State snapshot parsing remains backward compatible when `harness_path` is
  absent.
- Static dashboard rendering now includes `Workspace source: harness ...` for
  selected harness-backed workspaces.
- MIAOU selected-agent detail now includes `Source` and `Privileges` rows.
- The sidebar repeats the compact source line below the primary action.

## Validation

- `opam exec -- dune build @all @runtest`
- MIAOU headless harness-first test asserts:
  - `Source        harness .harness/harness.json`
  - `Privileges    reads 1 | writes 1`
- Repo-root tmux smoke shows the real 12-agent harness workspace with:
  - `Source        harness .harness/harness.json`
  - `Privileges    reads 11 | writes 11`

## Review Notes

The privilege summary is intentionally descriptive, not a new permission model:
it reports existing outgoing read/write ACL visibility as counts. Creation
authority still needs a first-class typed capability before TA should claim that
an agent can create agents in the UI.

Generated `index.json` remains unstaged.

## Next Loop Candidate

Add typed creation/connect capabilities to the TA model and render them in the
TUI, starting with `tech-lead` and `recruiter` as explicit privileged actors.
