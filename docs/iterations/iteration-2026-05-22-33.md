# Iteration 2026-05-22-33: Dashboard Viewport Height

## Brief

Loop 32 added structured action export. Loop 33 starts the Herdr-quality UI
baseline by making dashboard frame height explicit: CLI and future TUI callers
can request a terminal viewport height and receive a bounded frame with a clear
clipping marker instead of accidental overflow.

## Acceptance Criteria

- Add a pure viewport height abstraction shared by CLI and future TUI code.
- Keep existing unbounded dashboard rendering unchanged by default.
- Add `--height` to `ta`, `tactl dashboard render`, and
  `tactl dashboard render-socket`.
- Reject non-positive heights with a clear CLI error.
- Ensure clipped frames contain at most the requested row count and preserve the
  existing width contract.
- Cover pure viewport clipping, interaction rendering, state CLI, socket CLI,
  and `ta` entrypoint behavior.
- Run review, QA, live tmux smoke, commit, and push the loop branch.

## Result

Completed.

- Added `Dashboard_viewport`, with an abstract positive `height` type and a
  pure clipping function.
- `Dashboard_interaction.render` now accepts optional viewport height and clips
  after composing refresh/captured-at status lines with the dashboard frame.
- Added `--height` to `ta`, `tactl dashboard render`, and
  `tactl dashboard render-socket`.
- Added tests for invalid heights, exact row counts, clipping markers, and
  socket/state render wiring.
- Added direct socket-render bad-height regression coverage after independent
  review.
- Verified a live socket-backed tmux dashboard clips to 12 rows at 80 columns.
