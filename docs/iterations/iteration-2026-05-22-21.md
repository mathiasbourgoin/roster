# Iteration 2026-05-22-21: Dashboard Socket Refresh Loop

## Brief

Turn dashboard refresh intent into executable refresh behavior before wiring a
full MIAOU event loop. The previous loop added actor-scoped socket snapshots;
this loop lets keyboard refresh requests call that socket source, preserve the
current selection on success, and render stale state clearly when a refresh
fails.

## Acceptance Criteria

- Add explicit dashboard refresh status: fresh, refreshing, and stale with an
  error message.
- Add a typed socket refresh source that fetches and decodes dashboard snapshots.
- Make `tactl dashboard render-socket --key r` perform a socket refresh.
- Preserve selected workspace and agent across a successful refresh.
- Render the previous dashboard as stale when a refresh request fails after an
  initial snapshot was loaded.
- Keep deterministic CLI key replay for refresh requests.
- Cover refresh state, socket refresh decoding, and render-socket stale fallback
  with tests.
- Run real tmux QA for successful refresh and stale fallback.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added `Dashboard_interaction.refresh_status` with `Fresh`, `Refreshing`, and
  `Stale message`.
- Added `Dashboard_socket_refresh`, the effectful socket source the future
  MIAOU page can call on refresh ticks.
- `tactl dashboard render-socket --key r` now fetches a new dashboard model
  through the local socket, refreshes the interaction state, and preserves
  selection when the selected ids still exist.
- Refresh failures after the initial load render the previous snapshot with a
  `Refresh: STALE - ...` banner instead of aborting the frame.
- The local state dashboard path in `tactl` and `ta --state` also replays
  refresh keys deterministically through a state refresh closure.
- Added unit tests for refresh status rendering, socket refresh parsing errors,
  and CLI stale fallback.
- QA launched a disposable tmux workspace, rendered over a persistent socket
  with `--key r --key Down`, verified the live QA preview, then verified stale
  fallback with a one-shot socket server.

## Next Loop Candidate

Add a refresh cadence abstraction for the eventual MIAOU runner: a deterministic
tick source, refresh throttling, and user-visible last-refresh timestamps.
