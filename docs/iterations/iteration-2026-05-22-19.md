# Iteration 2026-05-22-19: Dashboard Interaction Boundary

## Brief

Move the dashboard from a static frame toward a MIAOU-ready interactive page
without adding a hard `miaou-tui` dependency before it is available in the TA
opam switch. This loop adds the state/update/view boundary MIAOU needs:
selection, key handling, refresh intent, and selected preview rendering.

## Acceptance Criteria

- Add dashboard interaction state for selected workspace, selected agent, and
  focused region.
- Add keyboard-style movement across agents and workspaces.
- Preserve selection across dashboard refreshes when ids still exist.
- Render the selected agent row and selected preview panel.
- Let `tactl dashboard render` and `ta --state` replay keys for deterministic
  QA.
- Cover interaction behavior and CLI key replay with tests.
- Run real tmux QA that selects a live agent preview through key replay.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Added `Dashboard_interaction`, a pure state/update/view layer modeled after
  MIAOU page boundaries.
- `Dashboard_model.render` now accepts an explicit selection and marks the
  selected workspace/agent while keeping line widths bounded.
- `tactl dashboard render --key KEY` and `ta --key KEY` replay dashboard keys
  before rendering; `--workspace` and `--agent` can set the initial selection.
- Added interaction tests for initial selection, agent movement, workspace
  movement, refresh preservation, and selected preview rendering.
- Added CLI tests for key replay through both `tactl dashboard render` and
  `ta --state`.
- QA launched a disposable two-agent tmux workspace, replayed `Down` to select
  the QA agent, and verified the selected live QA preview through both CLIs.

## Next Loop Candidate

Add a socket-backed dashboard refresh source so the interaction layer can swap
local state-file refresh for the local control plane before the concrete MIAOU
runner lands.
