# Iteration 2026-05-22-15: Socket-Supervised Launch

## Brief

Expose supervised workspace launch through the local TA socket so a UI or
orchestrator can ask the server to dry-run or start configured tmux workspaces
while the server owns the persisted state snapshot.

## Acceptance Criteria

- Add typed socket requests for `launch-dry-run` and `launch-start`.
- Add `tactl socket request --config ... launch-dry-run|launch-start` support.
- Reuse `Launch_plan`, `Launch_runtime`, and `Launch_state` rather than adding a
  second launch path.
- On `launch-start`, preflight against the served state snapshot, persist
  captured native pane ids, and clean up launch-created tmux sessions if state
  update fails.
- Require an authorized actor for socket `launch-start`.
- Cover the command path with tests and run a real tmux/socket launch QA smoke.
- Review, QA, document, commit, and push the loop branch.

## Result

Completed.

- Extended `Socket_protocol.request` with `Launch_dry_run` and `Launch_start`
  variants carrying a workspace config path and optional roster index path.
- Added socket API launch execution that loads config, optionally validates
  against a roster index, builds a launch plan, and delegates dry-run/start to
  the existing launch modules.
- `launch-start` now runs under the socket server's state ownership: it requires
  `--actor`, authorizes that actor against every launched agent, preflights the
  served snapshot, executes tmux launch, persists native pane ids with actor
  audit, and calls runtime cleanup if state persistence fails after tmux
  creation.
- Added `tactl socket request --config CONFIG [--roster-index INDEX]
  launch-dry-run` and `tactl socket request --config CONFIG --actor AGENT
  [--roster-index INDEX] launch-start`.
- Socket launch inputs are limited to regular files under a fixed size bound,
  and client-side relative config/roster paths are absolutized before sending to
  the socket server.
- Added CLI/socket regression coverage for launch dry-run, client-relative
  config paths with a server running from another CWD, missing `--config`,
  missing launch actor, unauthorized launch actor, and non-regular config paths.
- QA launched a disposable two-agent workspace through `socket serve --once`,
  verified the live tmux session existed, and confirmed state contained native
  pane ids plus actor-attributed pane attachment audit events.
- QA forced a state-persistence failure after tmux launch and verified the
  launch-created tmux session was cleaned up.

## Next Loop Candidate

Add cached runtime state collection for tmux sessions and panes. This should
give the future MIAOU/TUI dashboard a stable data model for live session status
and pane previews without polling tmux directly during render.
