# Roster Harness Determinism Audit — Synthesis Report

## 1. Executive Summary

Across 97 raw findings spanning the full roster pipeline (intake → question → research → spec → plan → implement → review → qa → ship → audit/investigate/init), the dominant pattern is that **roster's load-bearing gates are asserted by agent judgment rather than computed from artifacts**. The harness already names precise tunables (`max_function_lines: 50`, `min_user_stories: 2`, `0.7/0.3` fusion weights, `>365 days` staleness, `80%` coverage), defines clean fingerprint/traceability schemas, and even ships some deterministic scripts (`check-spec-trace.sh`, `check-kb-links.js`) — yet most of these are consumed by an agent reading prose and self-reporting compliance, so the same input can yield different verdicts run to run and a green checkmark can be written without the underlying gate having passed (classic Goodhart). The highest-leverage shifts are not new judgment, but **closing the loop on facts that are already mechanical**: (a) derive every GO/NO-GO and merge verdict from exit codes and the findings array instead of self-report; (b) replace "estimated"/"eyeball" size, duplication, and coverage measures with AST/CPD/coverage tooling against the thresholds already declared; (c) lint every brief/spec/findings artifact for structure, traceability IDs, and citation resolvability; and (d) make confidence/severity/risk labels a computed function of evidence class rather than a free-chosen number. Roughly 60% of the findings are pure bookkeeping or arithmetic masquerading as analysis and should be mechanized fully; the remaining ~40% correctly keep a creative/semantic core (adversarial review, root-cause synthesis, contradiction reasoning, blast-radius, priority) that must stay agentic — the recommendation everywhere is to mechanize *detection and proof-of-execution*, never the quality of the reasoning.

## 2. Themes

### Theme A — Verdicts asserted instead of derived from an exit-code/array oracle
**Pattern:** A status (`GO/NO-GO`, `VALIDATED`, `pass/fail`, `mergeable`, `COMPLETED`) is defined by a rule but written into the artifact by the agent, with nothing recomputing it. An agent can emit `GO` with an OPEN HIGH finding, or `COMPLETED` with a nonzero gate.
**Recurs in:** `roster-review.md` (Step 6 GO/NO-GO), `roster-qa.md` (Step 2/5 verdict table), `roster-ship.md` (BLOCK on NO-GO of review.json + qa.md), `roster-implement.md` (Step 6 self-filled gate checkboxes; Step 1/5 baseline-vs-final), `tech-lead.md` (merge policy), `qa.md` agent (pass/fail), `tdd-workflow.md` (RED/GREEN), `improvement-loop.md` (early-exit), `roster-run.md`/route detection (prose status scan).
**Deterministic + evidence replacement:** Compute the verdict as a pure function over machine-readable inputs — `status=GO iff count(findings where severity∈{CRITICAL,HIGH} and state==OPEN)==0`; `NO-GO iff any gate exit != 0`; `mergeable iff all Tier-1 exits 0 AND reviewer==approve AND open-critical==0`. Render gate tables from captured exit codes/parsed counts, and reject any self-reported status that mismatches the recomputed one. Make every brief carry a YAML frontmatter `status:` field so routing is `yaml-load + enum-check`, not a prose scan. Evidence: recomputed boolean vs declared status (mismatch = fail); aggregated exit codes.

### Theme B — "Estimated"/eyeballed size, duplication, complexity vs thresholds already declared
**Pattern:** The harness declares exact numeric thresholds (`max_function_lines`, `max_file_lines: 500`, `max_duplication_threshold: 0.15`, `ocaml_specialist_threshold: 50`, `auto_fix_threshold_lines`) but the measurement is an agent "estimate" via ad-hoc `grep + wc`, so the comparison is a gut call against a precise number.
**Recurs in:** `roster-audit.md` (Step 2 "estimated size", Step 3 DRY grep-guess), `code-quality-auditor.md` (function size, DRY), `architect.md` (excessive/deep/hotspot vs unwired tunables), `roster-implement.md` (Step 2 "lines of logic" routing), `roster-review.md` (Step 2 fix-first DRY + size gate).
**Deterministic + evidence replacement:** Wire the thresholds to real tooling — per-function line counts and nesting depth from tree-sitter/ctags/`lizard`/`gocyclo`/`eslint max-lines-per-function`; duplication from `jscpd`/PMD-CPD with the `≥5 lines`/`0.15` config; changed-line counts from `git diff --numstat`. Emit a finding only when measured value breaches the configured tunable, citing both numbers. Evidence: exact integer per function/file + duplication ratio vs the declared threshold; reproducible across runs.

### Theme C — Confidence / severity / risk picked by vibes → rubric keyed to evidence class
**Pattern:** Findings carry a free 1–5 confidence, a high/medium/low risk, or a CRITICAL/HIGH/MEDIUM severity with no rubric mapping evidence to the label, so the same finding lands differently across runs.
**Recurs in:** `roster-review.md` (confidence 1–5), `roster-investigate.md` (hypothesis probability), `improvement-loop-planner.md` (Confidence/Risk), `architect.md` (severity + overall risk), `qa.md` (defect severity), `roster-audit.md` (Step 7 buckets), `code-quality-auditor.md` (severity mapping), `spec-compliance-auditor.md` (Warning vs Info via "critical path"), `kb-agent`/`reviewer.md` (antipattern severity band).
**Deterministic + evidence replacement:** Define confidence as a constrained enum anchored to evidence type (`5 = reproduced by command`, `3 = static evidence`, `1 = pattern-match only`); define severity as a lookup over `(finding category × measured magnitude band × symbol-visibility/critical-path-glob)`. Hypothesis confidence = highest evidence state reached in `{untested → code-confirmed → refuted → observed}`. Allow a documented agent override that can only *raise* severity/risk, never silently lower it. Evidence: label is a function of recorded booleans/counts; the "never probably" rule becomes a denylist grep over evidence strings (`probably|likely|seems|looks`).

### Theme D — "Decide if X exists/matches/violates" → AST/grep/CPD query
**Pattern:** Detection tasks that are mechanically expressible (does this term appear? does this symbol exist? does this antipattern occur? is this invariant violated?) are done by manual reading, so coverage is non-exhaustive and depends on agent attention.
**Recurs in:** `ambiguity-auditor.md` (weasel-word list applied by eye, undefined-term extraction, stale/dead references, structured-field contradictions), `roster-audit.md` (naming variants, invariant anti-patterns), `code-quality-auditor.md` (invariant tracing), `reviewer.md` (lintable antipatterns via semgrep/ast-grep), `kb-agent.md` (numeric/bound contradictions), `spec-compliance-auditor.md`/`roster-spec-infer.md` (structured-signature conflicts).
**Deterministic + evidence replacement:** Ship the detection as a rule engine — weasel words as a vale/textlint/proselint config run over every KB file; dead refs via existence checks (`check-kb-links.js` already exists) + ctags symbol grep; undefined terms via regex candidate-extraction + glossary set-difference; antipatterns as semgrep/ast-grep rules over the diff; numeric/range invariant conflicts via bound-overlap check. The agent then *triages* the exhaustive hit list. Evidence: grep/tool hit count with `file:line` for every match; the "report every finding" rule is now actually met.

### Theme E — Structural/contract compliance left to "read before listing" honor-system
**Pattern:** Required sections, ID traceability, AC↔CHECK coverage, citation resolvability, minimum counts, and artifact-contract chaining are all stated as rules but enforced by the agent eyeballing markdown.
**Recurs in:** `roster-intake.md` (6 sections, slug regex, no-TBD, gate verification), `roster-plan.md`/`tech-lead.md`/`planner.md` (7-section brief + VALIDATED), `roster-spec.md` (min stories/GWT/challenges, AC↔CHECK 1:1, FR/AC traceability tags), `roster-research.md` (every finding has resolvable `file:line`), `roster-skill-evolve.md` (frontmatter + section validation, artifact-contract subset), `kb-agent.md` (no-blank sections), `roster-config.md` (type validation + YAML re-parse), `roster-investigate.md` (CONFIRMED rows cite an artifact).
**Deterministic + evidence replacement:** A small family of shared linters: `check-brief.sh` (required `##` headings non-empty + `status: VALIDATED`), a spec linter (count `### US-N`/`AC-N`/`C-N`, assert `≥` tunables, AC↔CHECK symmetric-diff empty, AC tags match `^<spec-slug>:AC-\d+$` and unique, no dangling `[US-N]` refs), a citation validator (every `path:line` resolves to an existing in-range line; each "unanswerable" carries a grep with 0 hits), a `validate-skill.sh` (frontmatter keys + required H2s), and `check-artifact-contracts.sh` (downstream `reads ⊆ upstream writes`). Evidence: linter exit code + per-rule pass/fail list; non-zero blocks the human gate. **Structure only — content adequacy stays human.**

### Theme F — Thresholds/triggers unquantified or unwired ("when in doubt", "fuzzy", "codebase size")
**Pattern:** A routing or gating decision is stated half-mechanically — one quantified signal paired with an unquantifiable companion ("≤20 words *and unambiguous*", "infers from question count *and codebase size*", "*fuzzy* or high-stakes").
**Recurs in:** `roster-run.md` (Express/Fast/Full), `roster-research.md` (depth auto, external-researcher spawn lexicon), `team.md` (diagnostic-interview trigger), `roster-skill-health.md` (clustering by "coherent theme", effort-mode), `kb-search.md` (0.7/0.3/0.3 magic numbers).
**Deterministic + evidence replacement:** Mechanize the computable sub-signals and keep judgment as an *override on top*, not a replacement: depth = `full if question_count>3 OR LOC>N OR files>M` (computed via `git ls-files | wc -l`); spawn external researcher on a defined trigger lexicon regex; force the diagnostic interview when high-stakes keywords (merge/rollback/deploy/governance) hit, never suppress it; cluster friction by `jq group_by(skill, suggestion_type, type)` then let the LLM merge free-text on top; calibrate `kb-search` weights/floor against a labeled fixture with precision@k/recall@k. **Critically, add a post-hoc contradiction catch** (e.g. Express chosen but `git diff --name-only` touched `kb/`/`specs/` → hard fail) rather than trying to pre-decide the mode mechanically.

### Theme G — Proof-of-execution unverifiable (did the mandatory pass actually run? was the gate actually run?)
**Pattern:** A mandatory step (adversarial pass, baseline gate, reproduction attempt, full suite) is trusted on the agent's word; a premature give-up is indistinguishable from a genuine result.
**Recurs in:** `roster-review.md` (Step 3b adversarial pass), `roster-implement.md` (Step 1 baseline capture), `roster-investigate.md` (Step 3 reproduction "not reproducible"), `qa.md` (Step 3 "ran the full suite"), `roster-init.md` (Mode B B1 "tests passing?" inferred).
**Deterministic + evidence replacement:** Require a recorded artifact as proof of execution: the adversarial sub-agent must emit a tagged result object (even "attempted, holds") whose presence is exit-checkable; capture baseline gate results to a file and diff against post-change; "not reproducible" must attach ≥1 recorded failed command + exit code; thoroughness gated on diff-coverage over changed lines; B1 records actual test exit code, linter violation count, `gh run list` CI status, and `grep -rcn TODO\|FIXME`. **This proves the pass *ran*, never that the reasoning was good.**

## 4. Keep-Agentic — Judgment That Must Not Be Mechanized

Reviewers explicitly flagged these as the over-correction hazard. Mechanizing them would create false confidence or reward vacuous artifacts.

- **Adversarial review reasoning** (`roster-review` 3b): breaking a change, spotting Goodhart/vacuous tests, blast-radius. Mechanize only *proof the pass ran*, never its quality. This is the core creative act of review.
- **Severity/priority *assignment* per finding** (review, audit, architect, qa, spec): the *rubric* removes drift, but the underlying call — is this actually a security bug? is a 600-line generated file fine while a 600-line god-object is not? P0 vs P1 business priority, falsifiability — stays human, with documented override that can only *raise* severity.
- **DIVERGE detection** (spec-compliance, audit): a tag proves a test *exists*, never that behavior *matches* the spec. DIVERGE must cite the specific behavioral difference.
- **Root-cause synthesis** (`investigate`): connecting evidence into a causal story, and choosing which hypothesis to test *first*, is irreducible. Only the post-test confidence label is evidence-derived.
- **Natural-language contradiction detection** (`ambiguity-auditor` 4, `kb-agent`): prose contradiction is not a diff. Only the structured-field subset (numbers, owners, enums, bounds) is mechanizable; do not over-formalize the KB.
- **"What vs how" mode selection** (`roster-run`): genuine judgment; deterministic layer only *catches contradictions after the fact* (Express that edited specs), never pre-decides the mode.
- **Adversarial-interview answer quality** (`roster-init` Mode A): judging substantive vagueness vs merely lacking a number is the legitimate creative core. Only question-count and metric-token presence are safe.
- **"One logical change" commit grouping** (`ship`): only the lint-able surface (format, length, casing) is mechanizable; grouping stays judgment.
- **Cross-module coupling / architecture-fit** (`architect`): not reducible to a line count; only size/length/nesting/duplication where thresholds exist.
- **Free-text friction "same workaround" merge** (`skill-health`): LLM merge stays a layer *on top* of deterministic field grouping; no exact-string matching on prose.
- **Whether a clone is *worth* refactoring / a duplicate is intentional** (audit, review): detection is mechanical; the refactor decision is judgment. Never let a line-count tool sneak a behavior change under the threshold; security/race/behavior-change rows never auto-fix.

A recurring guardrail across all of the above: **a schema-green or lint-green artifact must never be read as "correct"** — the linter proves the graph/structure is well-formed, not that the content is right; the human/substance gate stays.
