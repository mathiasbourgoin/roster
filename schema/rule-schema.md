# Rule Definition Schema

Rules are markdown files that define behavioral constraints for assistant sessions. Each rule lives in `rules/<category>/<name>.md` and is installed into the shared harness before being projected into runtime-specific locations.

## Required Frontmatter

```yaml
---
name: <string>               # Unique identifier (kebab-case)
description: <string>        # One-line summary of what this rule enforces
scope: <global|path:glob>    # Where the rule applies
category: <safety|style|workflow|language|governance|common>  # Classification
---
```

## Optional Frontmatter

```yaml
version: <semver>            # Version for tracking updates (e.g., 1.0.0)
applies_to: [<string>]       # Languages or frameworks this rule targets (e.g., ["ocaml", "python"])
```

## Categories

| Category     | Purpose                                          |
|--------------|--------------------------------------------------|
| `safety`     | Prevents harmful or irreversible actions         |
| `style`      | Code style, formatting, naming conventions       |
| `workflow`   | Process rules (commit style, PR flow, etc.)      |
| `language`   | Language-specific idioms and best practices       |
| `governance` | Human-gate, approval, and escalation rules       |
| `common`     | Shared rules not fitting other categories        |

## Directory Layout

```
rules/
  common/      # Cross-cutting rules
  governance/  # Approval, human-gate, escalation rules
  safety/      # Destructive-action guards
```

## Scope

- `global` — Applies to every file in the project.
- `path:<glob>` — Applies only when working on files matching the glob (e.g., `path:lib/**/*.ml`).

Runtime implementations may differ, but path-scoped rules should only apply when the active file matches.

## Body

The markdown body IS the rule content. Write it as direct instructions the runtime must follow. Be specific and testable — vague rules get ignored.

## Example

```markdown
---
name: sycophancy
description: Prevent agreement bias — challenge assumptions, flag uncertainties.
scope: global
category: safety
---

# Anti-Sycophancy

- When the user proposes a technical approach, evaluate it critically before agreeing. State at least one risk or alternative.
- Never say "Great idea!" or equivalent without substantive analysis.
- If you are uncertain about a claim, say so explicitly. Do not hedge with weasel words.
- When you change your position, explain what new information caused the change.
```

## Naming Convention

- File: `rules/<category>/<name>.md`
- Canonical shared location after install: `.harness/rules/<name>.md`
- Claude compatibility location: `.claude/rules/<name>.md`
- The `name` field must match the filename (without extension)

## Install Behavior

The canonical installer should place the rule in the shared harness and then generate any runtime-specific projections. Rules generated by the **governor** agent have `source: governor-generated` in the harness manifest. Custom rules added by the user have `source: custom`.
