# Spec — Injection-safe task-description heuristics

> Validation (standing user delegation, 2026-07-14): live vulnerability + fix empirically confirmed (old echo form creates canary, heredoc+printf form does not, HIT token preserved). SPI-1..4 binding. Snippets run via the agent Bash tool (bash); POSIX-conformant replacement — a strict improvement over the equally shell-dependent old form.

**Profile: minimal-freeze**

This is a trust-boundary `fix` (intake overruled Trust boundary no → yes): untrusted input (a task
description) crosses into shell execution. Three skill contracts tell the agent to substitute the
literal task description into `echo "<task>" | grep …` and run it; a description containing `$(…)`,
backticks, or `;cmd` is executed as code instead of matched as data. The fix replaces each site with
an injection-safe form — capture the description via a single-quoted heredoc into a variable, then
`printf '%s' "$desc" | grep …` — leaving the grep expression, flags, keyword list, and emitted HIT
token byte-identical so routing is unchanged.

No user-story or challenge ceremony applies to a minimal-freeze spec (roster-spec Trigger Check).
The requirements below are frozen **invariants**, numbered `SPI-N` (fresh non-colliding prefix),
range **SPI-1 … SPI-4**.

## The replacement snippet (canonical form)

Each of the three sites is rewritten from:

```bash
echo "<task description>" | grep -qiE "<keyword list>" && echo "<HIT token>"
```

to:

```bash
desc=$(cat <<'EOF'
<task description>
EOF
)
printf '%s' "$desc" | grep -qiE "<keyword list>" && echo "<HIT token>"
```

The single-quoted heredoc delimiter (`'EOF'`) disables all expansion in the body, so `$(…)`,
backticks, and `;cmd` in the pasted description are literal bytes; `printf '%s'` emits them verbatim
to grep. `<keyword list>`, the `-qiE` flags, and `<HIT token>` are copied unchanged per site, so
routing is identical.

## Invariants

- **SPI-1** [AC-1] — **Description is data, never executed.** At every heuristic site, a task
  description containing `$(cmd)`, backticks, or `;cmd` MUST be treated as literal bytes — never
  executed — and MUST still be scanned for keywords. Frozen by CHECK-1 (fail-closed-path): a payload
  like `x$(touch CANARY)` MUST NOT create CANARY when the snippet runs, and MUST still emit its HIT
  token when it contains a matching keyword.
- **SPI-2** [AC-2] — **Routing is byte-identical to the old form.** For every input, the keyword-match
  outcome and emitted HIT token MUST equal the old `echo`-interpolation form: a description that
  matched before still matches (same token), one that did not still emits nothing. Same keyword lists,
  same `-qiE` flags, same tokens. Frozen by CHECK-2 (authentic-success-path): a matching description
  reaches the real grep and produces the real routing token; a non-matching one emits nothing.
- **SPI-3** [AC-3] — **The sweep is complete.** No task-description-into-shell interpolation construct
  (`echo "<…>" | grep …`) MUST remain anywhere under `skills/`. Frozen by CHECK-3 (grep-clean).
- **SPI-4** [AC-4] — **Projections regenerated and consistent.** After the source edits and
  `bash scripts/sync-harness.sh`, all regenerated projections MUST be staged and identical to what the
  source regenerates, so `sync-harness --check` (last step of `npm test`) is green. Frozen by CHECK-4.

**Authentic-path pairing (roster-spec US-4).** CHECK-2 is the `(authentic-success-path)` — a valid,
keyword-bearing description reaches the real consumer boundary (the actual grep + HIT token that
downstream routing keys on) and is accepted. CHECK-1 is the `(fail-closed-path)` — a tampered/
malicious description is neutralized (command substitution does not execute) while still being
scanned. Both roles are covered by self-contained runnable checks.

## Runnable Checks

All checks honor the exit convention: `0` = check passes, `1` = assertion fired, `≥2` = error. Each
is a self-contained `bash -c` snippet (no external check file, no test-runner exit codes). The
injection payload is assembled at runtime by string concatenation; CANARY is a path under a
scratch/temp dir; payloads use `touch` (never `rm` — the repo denies `rm -rf`).

- **CHECK-1** [AC-1] **(fail-closed-path)** — injection payload is data, not code:

  ```bash
  bash -c '
  d=$(mktemp -d); can="$d/CANARY"
  # payload assembled at runtime: single-quoted segment keeps $(touch ...) literal, real path concatenated
  desc="deploy token x"'"'"'$(touch '"'"'"$can"'"'"')'"'"'
  hit=$(printf "%s" "$desc" | grep -qiE "auth|token|integrity" && echo TRUST_BOUNDARY_HIT)
  [ ! -e "$can" ] && [ "$hit" = "TRUST_BOUNDARY_HIT" ]; rc=$?
  rmdir "$d" 2>/dev/null; exit $rc'
  ```
  → expected exit `0`: CANARY absent (payload not executed) **and** HIT emitted (payload still
  scanned). The runtime-assembled literal `$(touch …)` stands in for a pasted description; the source
  form's single-quoted heredoc gives the same literalness at the paste site. Verified 2026-07-14: the
  old `echo` form creates CANARY and emits the token; the new form emits the token with **no** CANARY.

- **CHECK-2** [AC-2] **(authentic-success-path)** — routing parity through the real grep/token:

  ```bash
  bash -c '
  kw="auth|attest|evidence|authority|permission|token|signature|custody|integrity"
  m=$(desc="deploy token now"; printf "%s" "$desc" | grep -qiE "$kw" && echo TRUST_BOUNDARY_HIT)
  n=$(desc="refactor the widget layout"; printf "%s" "$desc" | grep -qiE "$kw" && echo TRUST_BOUNDARY_HIT)
  [ "$m" = "TRUST_BOUNDARY_HIT" ] && [ -z "$n" ]'
  ```
  → expected exit `0`: matching description emits the real HIT token, non-matching emits nothing —
  identical to the old form.

- **CHECK-3** [AC-3] — sweep is grep-clean:

  ```bash
  bash -c 'grep -rn "echo \"<" skills/ ; [ $? -eq 1 ]'
  ```
  → expected exit `0`: no `echo "<…>"` task-interpolation construct remains under `skills/`
  (`grep` exit 1 = no match = clean).

- **CHECK-4** [AC-4] — projections consistent:

  ```bash
  bash scripts/sync-harness.sh --check
  ```
  → expected exit `0`: no projection drift (the check the final step of `npm test` runs).

## Concerns

- **Non-bash runtime (raised by caller).** The three snippets are ` ```bash `-fenced and are executed
  by the agent's Bash tool, not a guaranteed-`bash` interpreter. The replacement uses only POSIX
  features — a quoted-delimiter here-document (`<<'EOF'`), command substitution `$(cat …)`, and
  `printf '%s'` — all mandated by POSIX.1, so the form behaves identically under `dash`/`ash`/`zsh`.
  Verified 2026-07-14 under `bash` and `/bin/sh` (both neutralize the payload and emit the token);
  `dash` was not installed on the dev host, so its identical behavior is argued from POSIX
  conformance rather than executed. If any runtime ever executed these via a shell lacking
  quoted-heredoc support (none in scope here), the freeze would need re-validation — but no such
  runtime exists in the roster projection set. The old `echo "<task>"` form was equally
  shell-dependent, so this is not a regression in portability, only in safety (strict improvement).
- **Version-bump granularity.** SPI-4 freezes projection consistency, not the specific version
  numbers; the two patch bumps (roster-run 1.10.2, roster-intake 1.3.1) are OQ-confirmed and are the
  only bumps intended. If a projection for a third skill changes, that is a signal the edit touched
  more than the three sites — SPI-3 and SPI-4 together catch it.
- **CHECK dual-role.** CHECK-1 and CHECK-2 exercise the `TRUST_BOUNDARY_HIT` site's keyword list; the
  `KEYWORD_HIT` site (roster-run.md:114) uses the identical replacement shape with its own list, so
  the same two invariants hold there by construction — the checks are representative, not exhaustive
  per site.
