## User Stories

### US-1: Versioned committed bundle (P0)
As a consumer project, I want the 14-file review-tool closure installed, upgraded, and removed as one versioned, hash-verified unit, so skill prose never points at tools that aren't there.
**Independent Test**: scratch install ‚Üí all 14 files present, shas match, manifest committed-style in place.

### US-2: Blocking preflight (P0)
As the pipeline, I want a missing/stale bundle to be a NOT-READY doctor verdict and a `blocked` skill stop with a runbook, so degradation is loud instead of a silent echo.
**Independent Test**: delete one bundle file ‚Üí doctor NOT-READY with runbook; roster-review guard records blocked/stale-install.

### US-3: Scratch consumer-install integration test (P0)
As the repo, I want install/upgrade/removal proven end-to-end in a throwaway consumer, so the distribution path can't silently rot.
**Independent Test**: the new test file passes in npm test; kills a mock file ‚Üí asserts no partial install.

### US-4: Machine-state gitignore provisioning (P1)
As a consumer, I want per-task machine state ignored by exact globs without hiding durable briefs.
**Independent Test**: init on a bare repo adds exactly the four globs; doctor warns on a tracked `*-state.json` without executing remediation.

## Amendments (v1.1.0 ‚ÄĒ plan-phase dual-voice review, 2026-07-13)

Both voices converged on the two structural fixes; eight more objections resolved. Amendments
override base FR/clarification text.

- **F-1 (O-2/V1, prose installer):** the entire install/upgrade/remove/verify state machine lives
  in ONE distributed bash script, `scripts/review-bundle-install.sh` (modes install|upgrade|
  remove|verify; --from-raw <url> | --from-checkout <dir>; --force). Bootstrap-fetched fresh each
  time (NOT manifest-listed ‚ÄĒ manifest stays the 14-file closure). Recruiter prose = two
  deterministic paste-executable lines; init-harness calls --from-checkout (warn-on-drift);

## Amendment (v1.3.0 ‚ÄĒ consumer schema fixtures, 2026-07-13)

Installing `schema/review-finding.schema.json` makes it visible to consumer repositories whose
validation gates auto-discover `schema/*.schema.json`. The bundle MUST therefore also install at
least one valid and one invalid fixture under
`tools/data-schema/fixtures/review-finding/{valid,invalid}`. These fixture entries are
sha-verified auxiliary bundle files, not JavaScript require-graph members. This keeps
auto-discovery gates non-tautological and prevents bundle installation from breaking an
otherwise healthy consumer.

The installed-consumer integration tier MUST load both installed fixtures through the installed
schema validator and prove that the valid record passes while the invalid record fails. Bundle
version 1.3.0 and roster-qa's `requires_review_bundle: ">=1.3.0"` make the fixtures mandatory for
QA readiness.
