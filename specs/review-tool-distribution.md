## Clarifications (challenge-resolution highlights; 22 challenges + 8 edge cases)

| Q | A |
|---|---|
| Committed or per-machine fetched? (C-11, load-bearing) | **Committed** — like skill projections. Teammate clones consistent; consumer CI can pass doctor; the committed manifest is the sole sentinel (no per-runtime sentinels — EC-2; no separate version file; resolves C-5's self-reference: manifest not in its own files[], removal deletes files minus shared plus the manifest). |
| Who forces manifest freshness / version bumps? (C-1/C-2) | Repo CI: data-driven manifest check (Check 6 refactor absorbed) fails on sha drift, on require()-edge escape (the 15th-file case), and on sha-change-without-version-bump vs git HEAD. |
| Compatibility owner (C-3/C-12/C-15) | Skill frontmatter `requires_review_bundle` (semver min); doctor reads it from installed projections (.claude primary, .agents fallback, drift → warn + max), compares to the local manifest — offline, no network (staleness = requirement violation, not latest-awareness). |
| Atomicity (C-4/EC-6) | Staging dir → verify all 14 shas → move → manifest LAST. Retry each fetch once, then abort; partial failure leaves staging only. No .bak files (C-19: tripwire has no allowlist). |
| Collisions / consumer-modified files (C-6) | Install refuses on paths matching neither old- nor new-manifest sha (abort + list; --force). Removal/upgrade skip modified files with warnings; missing files warn and continue (EC-7). |
| Wrapper dual ownership (C-7) | `xruntime-exec.sh` marked `shared: true`; removal skips shared; scratch test asserts roster-qa survives removal. Its backward-compatible `--prompt-file=<path>` API is part of the shared contract. |
| Extension system vs bespoke (C-8) | Bespoke by design: bash-only fetch must precede node trust, and `.harness` may not exist. Manifest conventions borrowed from the extension manifest for future convergence; a test asserts extension converge ignores bundle paths. |
| init-harness vs curl divergence (C-9) | Checkout path copies + WARNS on drift (dev tree is truth); curl path verifies shas (manifest is truth). Two guarantees, both documented. |
| Which installer (C-10) | Recruiter update-mechanism step (tools follow the skills that need them) + init-harness. install.sh explicitly does NOT fetch the bundle. |
| Hard-stop mechanics (C-13) | Prose-mechanical guard (bash existence test) → verdict `blocked`/`stale-install` + STOP with the runbook; never design-not-converging. Real gate = doctor NOT-READY. |
| Blast radius (C-14) | Bundle NOT-READY is deliberate fail-fast: a repo with roster-review ≥2.0 installed is using the review pipeline. Stated, not accidental. |
| Retrofit chicken-and-egg (C-22) | The guard fires only after new skills arrive; the same update that delivers them installs the bundle. Runbook (install.sh → /recruit update) in the NOT-READY message and the guard text. |
| briefs/ gitignore (C-20/C-21) | NEVER the whole directory (durable contracts live there). Exactly four machine-state globs; doctor prints (never runs) git rm --cached for the tracked-retrofit population. |
| Test fidelity (C-16/C-17/C-18) | file:// mock accepted with stated limits; both test tiers required (closure smoke + four functional invocations); upgrade fixture generated by running the REAL generator twice. |

file:// 404 exit-class difference; consumers ignoring scripts/ wholesale get a visibility warning

## User Stories

### US-1: Versioned committed bundle (P0)
As a consumer project, I want the 14-file review-tool closure installed, upgraded, and removed as one versioned, hash-verified unit, so skill prose never points at tools that aren't there.
**Independent Test**: scratch install → all 14 files present, shas match, manifest committed-style in place.

### US-2: Blocking preflight (P0)
As the pipeline, I want a missing/stale bundle to be a NOT-READY doctor verdict and a `blocked` skill stop with a runbook, so degradation is loud instead of a silent echo.
**Independent Test**: delete one bundle file → doctor NOT-READY with runbook; roster-review guard records blocked/stale-install.

### US-3: Scratch consumer-install integration test (P0)
As the repo, I want install/upgrade/removal proven end-to-end in a throwaway consumer, so the distribution path can't silently rot.
**Independent Test**: the new test file passes in npm test; kills a mock file → asserts no partial install.

### US-4: Machine-state gitignore provisioning (P1)
As a consumer, I want per-task machine state ignored by exact globs without hiding durable briefs.
**Independent Test**: init on a bare repo adds exactly the four globs; doctor warns on a tracked `*-state.json` without executing remediation.

## Amendments (v1.1.0 — plan-phase dual-voice review, 2026-07-13)

Both voices converged on the two structural fixes; eight more objections resolved. Amendments
override base FR/clarification text.

- **F-1 (O-2/V1, prose installer):** the entire install/upgrade/remove/verify state machine lives
  in ONE distributed bash script, `scripts/review-bundle-install.sh` (modes install|upgrade|
  remove|verify; --from-raw <url> | --from-checkout <dir>; --force). Bootstrap-fetched fresh each
  time (NOT manifest-listed — manifest stays the 14-file closure). Recruiter prose = two
  deterministic paste-executable lines; init-harness calls --from-checkout (warn-on-drift);
  doctor reuses `verify` mode; the scratch test executes the REAL script. One implementation,
  four call sites.
- **F-2 (O-1/V1, forced-bump baseline):** compare against `git merge-base origin/main HEAD`
  (fallback main → HEAD~1, mirroring check-leak-diff); manifest-absent-at-base tolerated
  (introducing commit); merge-base unresolvable (shallow clone) → skip with a loud warning
  (documented fail-open residual — sha-drift and require-escape checks still hold).
- **F-3 (O-3, preflight never runs Section 1):** the bundle check lives in **Section 2** as a
  gate-shaped probe (bundle = a detected gate whose missing/stale state → NOT-READY), so
  `/roster-doctor preflight` — the automated roster-run path — enforces it. Section 1 carries the
  detailed per-file report in full mode.
- **F-4 (O-4, `blocked` corrupts the ledger):** NO new verdict status. The guard is an
  **input-contract-style abort**: roster-review stops BEFORE writing any artifact (⛔ + runbook,
  no review.json, no ledger event) — the existing input-contract pattern. `stale-install` appears
  in the abort message (CHECK-6 grep target), never in a persisted status field. FR-144 amended.
- **F-5 (O-5, shared exempt):** doctor/`verify` checks presence + sha for ALL 14 files including
  shared; `shared: true` affects REMOVAL semantics only. Modified wrapper → NOT-READY with a
  restore/--force-reinstall message.
- **F-6 (O-6, EC-8 contradiction):** committed-model enforcement — in a git consumer repo,
  bundle files that are gitignored/untracked → NOT-READY reason "bundle not committed" with
  remediation (EC-8's warning-only clause superseded). `schema/` root colonization documented as
  accepted (same class as .claude//.agents/), per-file collision rules unchanged.
- **F-7 (O-7, headroom + YAML):** guard budget ≤80 pinned words (fenced bash is free); budget
  raise only per FR-120 process if compression genuinely fails. `requires_review_bundle: ">=1.0.0"`
  QUOTED in YAML; schema/skill-schema.md allowlist entry; roster-qa gets the frontmatter + a light
  wrapper existence check only (V1-Q4) + version bump.
- **F-8 (O-8, tripwire vs install.sh .bak idiom):** the no-.bak/no-stray invariant binds THE
  BUNDLE INSTALLER's own behavior and is asserted in the scratch fixture (which never runs
  install.sh). install.sh's pre-existing .bak idiom is out of scope, documented.
- **F-9 (O-9, Check-6 filter fragility):** the data-driven refactor ships WITH a fixture-driven
  unit test pinning each check's ✓/✗ independence (error-filter behavior), and fixes the latent
  ":247 escapes the filter" bug it absorbs.
- **F-10 (O-10, test cost/placement + Check-1 regex):** the scratch test follows the
  test-install.sh precedent — NOT in the npm-test chain; standalone `npm run test:review-bundle`
  plus a dedicated CI workflow step (⚠️ ci.yml edit — flagged for human visibility in the PR).
  The recruiter bundle step is placed after the blank line terminating the Skills-to-install
  list; the list-parse property is enforced live by check-pipeline-install.js's Check 1 (the
  recruiter install-list ↔ disk check), which runs on every `npm test` (CHECK-9) — not by a
  dedicated fixture. Generator/CI-check unit tests (CHECK-1) stay in the npm chain; only the
  heavy scratch integration test moves.

## Entities

- `BundleGenerator`: node script producing the manifest from the static closure list.
- `BundleDriftCheck`: repo-CI data-driven check (sha drift, closure escape, forced bump) — the Check-6 refactor.
- `BundlePreflight`: doctor Section-1 check feeding NOT-READY; `requires_review_bundle` frontmatter is its input.
- `ScratchConsumer`: the mkdtemp + mock-RAW integration fixture.
